## 1. 项目目标

把 `switch` 基座改造成一个独立桌面工具 `proxyswitch`，用于管理 177 机器上的代理开关，重点是：

- 一键开启 / 关闭代理
- 显示当前状态：开启、关闭、启动中、关闭中、异常
- 能明确知道当前是否走代理
- 保留开机自启能力
- 后续可继续扩展诊断能力

## 6. 现阶段开发方向

下一步只做 UI 和状态联动，不扩展新后端协议。

建议实现：

- 当前代理状态展示
- 开启 / 关闭代理按钮
- 代理连通性测试
- 自动启动开关
- 当前是否“正在走代理”的可视化提示

### 已完成的后续实现

- 前端已接入代理配置、自启、连通性检测与状态轮询。
- 已实现 SSH SOCKS 隧道的启动、停止、PID 状态持久化和端口监听检测。
- 只有隧道监听且应用代理配置指向 `socks5://127.0.0.1:7890` 时，界面才显示“已开启”。
- 停止时会验证 PID 对应的仍是 ProxySwitch 启动的 SSH 隧道，避免误杀其他进程。
- GitHub 诊断会经 SOCKS 隧道访问 GitHub API，并只读取 Git 全局代理是否已配置及是否指向本地隧道。
- “开机启动应用”与“应用启动后自动连接代理”已拆分为两个独立设置；后者会在启动后异步建立隧道并应用代理，失败不会阻塞主界面。

## 7. 推荐的页面结构

主界面尽量简单：

- 状态卡片
- 开启按钮
- 关闭按钮
- 测试按钮
- 自动启动开关
- 错误提示区域

如果状态变化，要能立即反映：

- `off`
- `starting`
- `on`
- `stopping`
- `error`

## 8. 继续开发时的优先顺序

1. 先确认前端是否已经切到 `proxyswitch` 的主界面
2. 把前端按钮接到现有后端命令
3. 把“当前状态”做成实时刷新
4. 把“开启 / 关闭 / 测试”三件事跑通
5. 再补自启开关
6. 最后再做清理和打包

## 9. 需要重点检查的文件

建议优先看这些文件：

- `src/App.tsx`
- `src-tauri/src/lib.rs`
- `src-tauri/src/commands/global_proxy.rs`
- `src-tauri/src/commands/settings.rs`

## 10. 注意事项

- 不要一次性大改前后端，容易把当前可用部分弄坏
- 不要删除旧文件，先放进 `tmp/`
- 如果代理开启后 GitHub 不通，优先排查：
  - 代理是否真的启动
  - 本地端口是否监听
  - 系统代理是否被错误覆盖
  - Git 全局代理是否还指向旧地址
  - VPN 与代理是否互相冲突

这个项目现在的核心不是“再做一个代理工具”，而是：
